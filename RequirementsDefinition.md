# JRA競馬予想アプリケーション 要件定義書

## 1. プロジェクト概要

### 1.1 プロジェクト名
JRA競馬予想アプリケーション

### 1.2 目的
JRAの競馬データをスクレイピングし、機械学習モデルを用いて勝馬予想を行い、Webインターフェースで予想結果を提供するシステムを構築する。

### 1.3 対象ユーザー
- 競馬愛好家
- データに基づいた予想を求める利用者
- 競馬データの分析に興味がある研究者

### 1.4 プロジェクトスコープ
- JRAの公式競馬データの収集・蓄積
- 機械学習による勝馬予想モデルの構築
- Webアプリケーションによる予想結果の提供
- REST APIによるデータアクセス

## 2. 機能要件

### 2.1 データ収集機能

#### 2.1.1 レースデータのスクレイピング
**優先度**: 高

**概要**: JRAウェブサイトから競馬のレース情報を自動収集する。

**詳細要件**:
- **レースカレンダーの取得**
  - 指定日付のレース開催情報を取得
  - 開催競馬場、レース数の情報を取得

- **レース詳細情報の取得**
  - レースID、レース名
  - 開催日、競馬場、レース番号
  - 距離、芝/ダート、馬場状態、天候
  - レースクラス（G1、G2、G3、OP、条件戦など）
  - 賞金額

- **出走馬情報の取得**
  - 馬名、馬ID、性別、生年月日
  - 騎手名、騎手ID
  - 調教師名、調教師ID、所属厩舎
  - 枠番、馬番
  - 斤量、馬体重、馬体重増減
  - 朝オッズ

- **レース結果の取得**
  - 着順、走破タイム
  - 着差
  - 確定オッズ、人気
  - コーナー通過順位
  - レースコメント
  - 払戻金（単勝、複勝、馬連など）

- **馬の詳細プロフィール取得**
  - 血統情報（父馬、母馬）
  - 過去の戦績
  - 獲得賞金

**制約条件**:
- スクレイピング間隔: 2-5秒（サーバー負荷を考慮）
- User-Agent設定の適切な管理
- JRA利用規約の遵守
- robots.txtの尊重

#### 2.1.2 データ保存
**優先度**: 高

**概要**: 収集したデータをデータベースに保存する。

**詳細要件**:
- リレーショナルデータベースへの正規化された形式での保存
- 重複データの防止（ユニークID管理）
- データの更新（レース結果の追加など）
- タイムスタンプ管理（作成日時、更新日時）

### 2.2 データ処理・分析機能

#### 2.2.1 特徴量エンジニアリング
**優先度**: 高

**概要**: 機械学習モデルの入力として使用する特徴量を生成する。

**詳細要件**:
- **馬に関する特徴量**
  - 過去の成績（勝率、連対率、複勝率）
  - 距離別成績
  - 馬場状態別成績
  - コース別成績
  - 休養明けフラグ
  - 馬齢、性別
  - 馬体重、馬体重増減

- **騎手に関する特徴量**
  - 騎手の勝率、連対率
  - 当該コースでの成績
  - 当該馬との相性

- **調教師に関する特徴量**
  - 調教師の勝率
  - 所属厩舎の実績

- **レースに関する特徴量**
  - レースクラス
  - 距離、芝/ダート
  - 馬場状態、天候
  - 賞金額

- **オッズ情報**
  - 朝オッズ
  - オッズ変動

- **血統情報**
  - 父馬、母馬の実績
  - 種牡馬ランキング

#### 2.2.2 データクレンジング
**優先度**: 高

**詳細要件**:
- 欠損値の処理
- 異常値の検出と処理
- データ型の統一
- カテゴリカルデータのエンコーディング

### 2.3 機械学習モデル機能

#### 2.3.1 モデル訓練
**優先度**: 高

**概要**: 過去のレースデータを用いて勝馬予想モデルを訓練する。

**詳細要件**:
- **使用するアルゴリズム**
  - RandomForest
  - XGBoost
  - LightGBM
  - ニューラルネットワーク（将来的に）

- **モデル評価指標**
  - 的中率（Accuracy）
  - 適合率、再現率、F1スコア
  - 回収率（ROI）
  - AUC-ROC

- **訓練プロセス**
  - データの分割（訓練:テスト = 8:2）
  - クロスバリデーション
  - ハイパーパラメータチューニング
  - モデルの保存とバージョン管理

#### 2.3.2 予想生成
**優先度**: 高

**概要**: 訓練済みモデルを用いて未来のレースの予想を生成する。

**詳細要件**:
- 予想着順の出力
- 勝率（win probability）の算出
- 信頼度スコアの付与
- 複数モデルのアンサンブル予想
- 予想結果のデータベース保存

#### 2.3.3 モデルの評価と更新
**優先度**: 中

**詳細要件**:
- 予想精度の追跡
- 実際の結果との比較
- モデルの定期的な再訓練
- A/Bテスト機能

### 2.4 Webアプリケーション機能

#### 2.4.1 トップページ
**優先度**: 高

**概要**: アプリケーションのホーム画面を提供する。

**詳細要件**:
- 今後7日間の開催予定レース一覧（最大10件）
- 最近完了したレース一覧（最新5件）
- ナビゲーションメニュー
- サイトの簡単な説明

#### 2.4.2 レース一覧ページ
**優先度**: 高

**概要**: レース情報を一覧表示する。

**詳細要件**:
- フィルタリング機能
  - 日付指定
  - 競馬場選択
  - ステータス（開催予定/完了）
- ソート機能
- ページネーション
- レース詳細へのリンク

#### 2.4.3 レース詳細ページ
**優先度**: 高

**概要**: 個別レースの詳細情報を表示する。

**詳細要件**:
- レース基本情報（日時、競馬場、距離、馬場状態など）
- 出走馬一覧
  - 馬名、騎手、調教師
  - 枠番、馬番
  - 斤量、馬体重
  - オッズ
- レース結果（完了済みの場合）
  - 着順、走破タイム
  - 払戻金

#### 2.4.4 予想結果ページ
**優先度**: 高

**概要**: 機械学習モデルによる予想結果を表示する。

**詳細要件**:
- 予想一覧ページ
  - 予想が生成されているレース一覧
- レース別予想詳細
  - 予想着順
  - 勝率
  - 信頼度スコア
  - 使用モデル名とバージョン
- 予想の根拠（特徴量の重要度など）
- 複数モデルの予想比較

#### 2.4.5 予想精度ページ
**優先度**: 中

**概要**: モデルの予想精度を統計的に表示する。

**詳細要件**:
- 総予想回数
- 的中率
- モデル別の精度比較
- 期間別の精度推移グラフ
- 回収率の表示

#### 2.4.6 About/情報ページ
**優先度**: 低

**詳細要件**:
- アプリケーションの説明
- 使い方ガイド
- 免責事項
- 法的配慮の記載

### 2.5 REST API機能

#### 2.5.1 レースデータAPI
**優先度**: 中

**エンドポイント**:
- `GET /api/races` - レース一覧取得
- `GET /api/races/<race_id>` - レース詳細取得
- `GET /api/races/<race_id>/entries` - 出走馬一覧取得
- `GET /api/races/<race_id>/results` - レース結果取得

#### 2.5.2 予想API
**優先度**: 中

**エンドポイント**:
- `GET /api/predictions/race/<race_id>` - レース予想取得
- `GET /api/predictions/accuracy` - 予想精度統計取得

#### 2.5.3 馬・騎手データAPI
**優先度**: 低

**エンドポイント**:
- `GET /api/horses/<horse_id>` - 馬の詳細情報
- `GET /api/jockeys/<jockey_id>` - 騎手の詳細情報
- `GET /api/trainers/<trainer_id>` - 調教師の詳細情報

**API仕様**:
- レスポンス形式: JSON
- 認証: 将来的にAPIキーまたはトークンベース認証
- レート制限: 1秒あたり10リクエスト（将来的に）

### 2.6 管理・運用機能

#### 2.6.1 スケジューラー機能
**優先度**: 中

**詳細要件**:
- 定期的なデータスクレイピング（1日1回、深夜1時など）
- 自動モデル訓練（週次または月次）
- 開催予定レースの自動予想生成

#### 2.6.2 ロギング機能
**優先度**: 高

**詳細要件**:
- アプリケーションログの記録
- エラーログの記録
- スクレイピングログ
- ログレベルの設定（DEBUG, INFO, WARNING, ERROR）
- ログのローテーション

#### 2.6.3 エラーハンドリング
**優先度**: 高

**詳細要件**:
- 404エラーページ
- 500エラーページ
- スクレイピングエラーのリトライ機能
- データベースエラーのロールバック

## 3. 非機能要件

### 3.1 パフォーマンス要件

- **Webページのロード時間**: 3秒以内
- **API応答時間**: 500ms以内
- **データベースクエリ**: インデックスを使用した最適化
- **同時接続数**: 100ユーザーまで対応（初期）

### 3.2 スケーラビリティ

- データベースの水平スケーリング対応
- アプリケーションサーバーの複数インスタンス対応（将来的に）
- キャッシュ機構の導入（Redis等、将来的に）

### 3.3 セキュリティ要件

- **SQL Injection対策**: ORMの使用、パラメータ化クエリ
- **XSS対策**: テンプレートエンジンの自動エスケープ
- **CSRF対策**: Flask-WTFの使用
- **環境変数管理**: .envファイルでの秘密情報管理
- **HTTPS**: 本番環境ではHTTPS必須

### 3.4 可用性

- **稼働率**: 99%以上（本番環境）
- **バックアップ**: データベースの日次バックアップ
- **エラーリカバリー**: 自動リトライ機能

### 3.5 保守性

- **コード品質**: PEP 8準拠
- **コードフォーマット**: Blackによる自動フォーマット
- **リント**: flake8によるコードチェック
- **テストカバレッジ**: 80%以上
- **ドキュメント**: 主要モジュールのdocstring記述

### 3.6 互換性

- **ブラウザ対応**: Chrome, Firefox, Safari, Edge（最新版）
- **モバイル対応**: レスポンシブデザイン
- **Python バージョン**: Python 3.9以上

### 3.7 法的・倫理的要件

- **JRA利用規約の遵守**: スクレイピング時の利用規約遵守
- **著作権**: データの商用利用時はJRAへの事前確認
- **免責事項の明記**: 予想結果の正確性を保証しないことを明示
- **個人情報保護**: ユーザー情報を収集する場合はプライバシーポリシーの策定
